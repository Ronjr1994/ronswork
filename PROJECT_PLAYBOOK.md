# Client-Ready Portfolio Playbook

## Purpose

Use this pattern to turn a personal portfolio into a credible service experience that helps a client understand the offer, verify proof, and start a well-scoped conversation.

## Core sequence

1. **Positioning first** — state the problem solved before listing skills.
2. **Trust early** — surface public work, feedback, labels, and disclosures near the top.
3. **Proof with context** — every case study includes challenge, role, output, and outcome.
4. **Process clarity** — show how an engagement moves from diagnosis to handoff.
5. **Guided conversion** — replace a vague contact form with a structured brief.
6. **Privacy by default** — avoid collecting sensitive data and keep local saving optional.
7. **Accurate automation claims** — describe browser-based guidance as guided automation, not external AI.
8. **Responsive end states first** — design the final desktop and mobile layout before animating.
9. **Production QA** — validate links, scripts, assets, accessibility, and deployment files.
10. **Reusable documentation** — preserve the system, not only the final design.

## Delivery stages

### Stage 1 — Strategy
Define the audience, pain points, promise, services, proof, conversion goal, and primary CTA.

### Stage 2 — Visual system
Lock typography, palette, spacing, corner treatment, strokes, imagery, and responsive breakpoints.

### Stage 3 — Interaction
Build navigation, motion, tabs, galleries, reduced-motion behavior, and stable animation end states.

### Stage 4 — Conversion
Add the project brief, service matching, privacy warning, copy/email fallback, and optional secure endpoint.

### Stage 5 — Production
Optimize assets, add metadata, privacy information, validation scripts, sitemap, robots file, and deploy.

## Non-negotiable QA gates

- Every public claim is verifiable or clearly labeled as a concept.
- No patient information, passwords, or confidential files are requested.
- The site works without an AI API.
- The primary CTA remains usable when direct submission is unavailable.
- The hero end state has no overlap on desktop, laptop, tablet, Android, or iOS.
- Keyboard users can open, navigate, and close dialogs.
- Local draft saving is opt-in and clearable.
- All local references exist before packaging.
- The deployment ZIP has `index.html` at the root.
